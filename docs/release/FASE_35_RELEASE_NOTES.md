# FASE 35 - Sigilo Vivo: Foundations - Release Notes

**Date:** 2026-03-25
**Version:** 0.35.0
**Status:** Completed

## Executive Summary

FASE 35 delivered the foundational layer of Sigilo Vivo — the live graph visualization system for CCT applications. Four subphases define the canonical model for route topology metadata, the web-focused sigil SVG style, the `.ctrace` file format with a CLI viewer, and a declarative manifest system for external frameworks to publish their topology without compiler coupling.

Sigilo Vivo transforms the existing sigil graph (CCT's structural visualization) into a live, queryable, and trace-correlated tool for understanding runtime behavior.

## Scope Closed

Delivered subphases:

- **35A**: Structural route metadata — canonical model for endpoints, groups, middleware, handlers
- **35B**: Navigable route SVG — web-focused sigil style with HTTP semantics
- **35C**: `.ctrace` format and viewer CLI — trace artifact and terminal/SVG tooling
- **35D**: External framework manifests — topology declaration without compiler coupling

## Key Deliverables

### 35A — Structural Route Metadata

New entities in the sigil model:

- `SigilRoute` — named endpoint with HTTP method, path pattern, handler reference
- `SigilGroup` — prefix group containing child routes and middleware
- `SigilMiddleware` — named middleware with before/after attachment points
- `SigilHandler` — callable unit bound to a route

Serialization is additive: metadata is embedded in the existing `.sigil` format without breaking the base sigil reader.

CLI:

```bash
cct sigilo routes list my_app.sigil
cct sigilo routes show GET /users my_app.sigil
```

### 35B — Navigable Route SVG

Web-focused sigil rendering:

- Nodes represent route groups and endpoints
- Edges represent middleware chains and handler bindings
- HTTP method badges on endpoint nodes (color-coded by method)
- Static SVG output compatible with `file://` protocol (no external dependencies)

Generated by:

```bash
cct --sigilo-only my_app.cct          # automatic generation
cct sigilo render my_app.sigil        # explicit render from .sigil
```

### 35C — `.ctrace` Format and Viewer CLI

`.ctrace` is the canonical trace artifact format:

- JSON Lines, one span object per line
- Fields: `trace_id`, `span_id`, `parent_id`, `name`, `category`, `start_us`, `end_us`, `attrs`
- Tolerant parser (ignores unknown fields) and strict parser (rejects unknown fields)

CLI tooling:

```bash
cct sigilo trace view request.ctrace           # terminal tree viewer
cct sigilo trace view --flat request.ctrace    # flat timeline
cct sigilo trace export request.ctrace out.svg # SVG export
```

### 35D — External Framework Manifests

Declarative manifest format (`.system.sigil`) for frameworks:

- Declare routes, pages, middleware, and background tasks
- No compiler dependency: frameworks write the manifest file directly
- Sigil renderer reads `.system.sigil` to augment the application sigil with framework topology

Format sections: `[routes]`, `[middleware]`, `[pages]`, `[tasks]`, `[metadata]`

CLI:

```bash
cct sigilo manifest validate civitas.system.sigil
cct sigilo manifest merge app.sigil civitas.system.sigil merged.sigil
```

## Design Decisions

### Sigil as the Ground Truth

Route metadata, trace correlation, and framework topology all anchor to the sigil rather than introducing a separate topology store. The sigil is the single structural artifact; Sigilo Vivo layers live semantics on top of it.

### `.ctrace` Is File-First

Traces are files, not streaming RPC calls. This enables post-mortem analysis, CLI tooling, and comparison across environments without requiring a live server.

### External Manifests Without Compiler Coupling

FASE 35D avoids requiring the CCT compiler to understand framework-specific concepts. Frameworks emit manifests; the sigil tools consume them. This keeps the compiler core stable while enabling rich topology visualization in the ecosystem.
