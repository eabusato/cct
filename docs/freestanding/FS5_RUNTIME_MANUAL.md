# CCT Freestanding FS-5 Runtime Manual

## Scope

FS-5 closes the first freestanding HTTP-server block:

- FS-5A: connection accept loop and `cct/http_server_fs`
- FS-5B: HTTP/1.1 parser and `cct/http_parser_fs`
- FS-5C: HTTP response builder and `cct/http_response_fs`
- FS-5D: HTTP router, CCT handlers, and `cct/http_router_fs`

The concrete closure gate is `G-FS5`: the booted system serves HTTP on bare metal and returns HTML generated by CCT for `GET /`.

## Canonical Modules

### `cct/http_server_fs`

Provides the CCT-facing server lifecycle:

- `http_server_fs_init(port)`
- `http_server_fs_accept(timeout_ms)`
- `http_server_fs_read()`
- `http_server_fs_send(data, len)`
- `http_server_fs_send_str(text)`
- `http_server_fs_close()`
- `http_server_fs_get_request(buf, max_len)`
- `http_server_fs_req_count()`
- `http_server_fs_req_len()`

Current operational model:

- one active connection at a time
- poll-based accept/read loop
- header timeout: `5000 ms`
- body timeout: `10000 ms`
- auto-relisten on close

### `cct/http_parser_fs`

Provides parsed request access without host services:

- request line access: method, path, query, version
- header enumeration and lookup
- method and path predicates
- body length and body copy
- console dump helper for on-screen diagnostics

Current parser limits:

- fixed-size request fields
- static header table
- no chunked transfer decoding
- body size depends on the raw request buffer gathered by FS-5A

### `cct/http_response_fs`

Provides the HTTP response builder surface:

- direct send helpers: `html`, `json`, `text`, `404`, `405`, `500`
- staged builder path:
  - `http_response_fs_begin(status)`
  - `http_response_fs_add_header(name, value)`
  - `http_response_fs_finish(content_type, body)`
  - `http_response_fs_len()`

Current wire contract:

- status line plus canonical reason text
- `Content-Length`
- `Connection: close`
- optional extra headers inserted before the body

### `cct/http_router_fs`

Provides the static route table surface:

- router init
- route add with exact or prefix matching
- 404 handler id registration
- dispatch by `(method, path)`

Current matching contract:

- exact match for concrete paths such as `/` and `/status`
- prefix match for subtree handlers such as `/api`
- dispatch result is a numeric `handler_id` chosen in CCT and resolved by the boot application

## Compiler Contracts

Host-profile gating now rejects these modules outside `--profile freestanding`:

- `cct/http_server_fs`
- `cct/http_parser_fs`
- `cct/http_response_fs`
- `cct/http_router_fs`

Semantic validation and C generation now recognize the FS-5 builtins for:

- HTTP server init/accept/read/send/close
- request parse and request-field/header/body access
- response build/send
- route registration and dispatch

The generated `.cgen.c` objects compile cleanly with `i686-elf-gcc` and link into the bootable kernel.

## Boot Integration Contract

The active FS-5 entry is `../grub-hello/src/civitas_boot_fase5.cct`.

Actual boot flow:

1. `kernel_main()` still prints the initial C banner and initializes the freestanding heap
2. low-level setup still includes GDT, PIC, IDT, keyboard, timer, and PCI support
3. `civitas_boot_fase5()` reuses the FS-2 presentation step through `civitas_boot_fase2()`
4. interrupts are enabled, a short timer-backed delay runs, and the screen is cleared
5. PCI and RTL8139 bring-up are reused from the FS-4 networking base
6. the HTTP server, parser, response builder, and router are initialized on port `80`
7. the FS-5 screen shows the HTTP routes and live gate instructions
8. the steady-state loop accepts a request, parses it, dispatches to a CCT handler, sends the response, closes the connection, and relistens

The integrated `../grub-hello/Makefile` now compiles:

- `src/http_server.c`
- `src/http_parser.c`
- `src/http_response.c`
- `src/http_router.c`
- `src/civitas_boot_fase5.cct`

The compatibility artifact names remain stable:

- `build/civitas_boot.cct`
- `build/civitas_boot.cgen.c`
- `build/civitas_boot.o`

## Gate Contract

The booted application installs these concrete handlers:

- `GET /` → HTML generated by CCT
- `GET /status` → JSON generated by CCT
- `GET /api/*` → JSON generated by CCT through a prefix route
- `POST /echo` → text response generated by CCT from parsed request metadata
- default → HTML 404 page generated by CCT

`G-FS5` is considered closed when:

- the VGA screen shows the FS-5 HTTP server state
- the booted image accepts HTTP requests on port `80`
- `GET /` returns HTML generated by the CCT handler path
- `GET /status` returns JSON system status


Boot integration:

```bash
cd ../grub-hello && make iso
```

```bash
python3 tests/support/freestanding_fs5_gate_probe.py \
  --qemu-bin qemu-system-i386 \
  --iso ../grub-hello/grub-hello.iso
```

Expected evidence:

- VGA boot screen shows `=== FS-5: HTTP Freestanding ===`
- `GET /` returns `HTTP/1.1 200 OK` plus the CCT-generated HTML marker
- `GET /status` returns JSON with `service`, `ip`, `uptime_ms`, and request counters
- `GET /api/info` proves prefix-route dispatch
- `POST /echo` proves request-body metadata reaches the CCT handler path

## Limits

- the HTTP service remains single-connection and synchronous
- HTTP/1.1 keep-alive is not supported; the runtime closes after every request
- no chunked request decoding is claimed in FS-5
- no TLS, compression, or dynamic routing tables are claimed in FS-5
